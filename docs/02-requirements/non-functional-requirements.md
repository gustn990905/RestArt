# Non-Functional Requirements

## 1. 문서 목적

이 문서는 RestArt 서비스의 비기능 요구사항을 정리한다.

비기능 요구사항은 보안, 성능, 확장성, 유지보수성, 파일 관리, API 안정성, 이미지 처리 품질, Unity/AR 프로젝트 관리 기준을 포함한다.

---

## 2. 보안 요구사항

### 2.1 민감 정보 관리

다음 정보는 저장소에 포함하지 않는다.

- `.env`
- DB password
- AWS access key
- AWS secret key
- `.pem` 파일
- 운영 DB host
- 운영 계정 정보
- 실제 사용자 개인정보
- 실제 운영 DB dump

### 2.2 환경변수 관리

환경변수는 `.env.example` 형태로만 제공한다.

예상 환경변수는 다음과 같다.

- `DB_HOST`
- `DB_PORT`
- `DB_NAME`
- `DB_USER`
- `DB_PASSWORD`
- `AWS_ACCESS_KEY_ID`
- `AWS_SECRET_ACCESS_KEY`
- `S3_BUCKET_NAME`

---

## 3. API 안정성

백엔드 API는 웹, 모바일 앱, AI 추천 모델, DB와 연결되므로 안정적인 요청·응답 구조가 필요하다.

요구사항은 다음과 같다.

- API 요청과 응답 형식을 문서화해야 한다.
- 이미지 업로드 실패 시 명확한 오류 메시지를 반환해야 한다.
- 추천 결과가 없을 경우 빈 목록과 안내 메시지를 반환해야 한다.
- DB 연결 실패 시 서버 오류가 노출되지 않도록 처리해야 한다.
- 요청 파라미터 검증을 수행해야 한다.

---

## 4. 성능 요구사항

RestArt는 이미지 업로드, 색상 추출, 추천 계산을 포함하므로 처리 성능이 중요하다.

요구사항은 다음과 같다.

- 이미지 업로드 후 분석 결과가 과도하게 지연되지 않아야 한다.
- 추천 결과는 사용자가 기다릴 수 있는 시간 안에 반환되어야 한다.
- 대용량 이미지의 경우 압축 또는 리사이징을 적용할 수 있어야 한다.
- DB 조회는 필요한 컬럼과 인덱스를 기준으로 최적화해야 한다.
- 추천 계산은 배치 처리와 실시간 처리를 구분할 수 있어야 한다.

---

## 5. 이미지 처리 품질

이미지 분석 기능은 추천 품질에 직접 영향을 준다.

요구사항은 다음과 같다.

- 이미지 형식을 제한해야 한다.
- 이미지 크기 제한을 설정해야 한다.
- 흐릿하거나 너무 어두운 이미지에 대한 처리 기준이 필요하다.
- 공간 이미지 업로드 시 촬영 가이드를 제공할 수 있어야 한다.
- 색상 추출 결과는 DB에 재사용 가능한 형태로 저장되어야 한다.

---

## 6. DB 확장성

작품, 사용자, 추천 결과, 리플릿, 전시 데이터가 증가할 수 있으므로 DB는 확장 가능한 구조여야 한다.

요구사항은 다음과 같다.

- 작품과 색상 데이터는 분리 저장할 수 있어야 한다.
- 작품과 감성 데이터는 다대다 관계를 고려할 수 있어야 한다.
- 추천 결과는 사용자별 이력으로 저장될 수 있어야 한다.
- 리플릿 생성 결과는 재조회 가능해야 한다.
- 거래 데이터는 작품 상태와 연결되어야 한다.

---

## 7. 유지보수성

기능이 여러 영역에 걸쳐 있으므로 모듈 분리가 중요하다.

요구사항은 다음과 같다.

- API 라우터는 기능별로 분리한다.
- 추천 알고리즘은 백엔드 라우터와 분리한다.
- DB 연결 설정은 별도 파일로 분리한다.
- 색상 추출, 감성 분석, 추천 계산은 독립 모듈로 관리한다.
- 웹, 앱, Unity/AR 코드는 각 앱 폴더에서 분리 관리한다.

---

## 8. Unity/AR 파일 관리

Unity 프로젝트는 파일 용량과 자동 생성 파일이 많으므로 별도 관리 기준이 필요하다.

저장소에 포함하지 않는 파일은 다음과 같다.

- `Library/`
- `Temp/`
- `Obj/`
- `Build/`
- `Builds/`
- `Logs/`
- `UserSettings/`
- `.vs/`
- `*.csproj`
- `*.sln`

필요한 파일만 저장한다.

- `Assets/`
- `Packages/`
- `ProjectSettings/`
- 주요 C# scripts
- 시연용 scene
- README

대용량 영상이나 모델 파일은 필요 시 Git LFS 사용을 검토한다.

---

## 9. 접근성 및 사용자 경험

RestArt는 예술 작품 탐색과 구매 경험을 단순하게 제공해야 한다.

요구사항은 다음과 같다.

- 작품 추천 과정은 사용자가 이해할 수 있어야 한다.
- 추천 결과에는 추천 이유를 제공할 수 있어야 한다.
- AR 설치 기능은 직관적인 조작 방식을 가져야 한다.
- 모바일 리플릿은 SNS 공유에 적합해야 한다.
- 작품 상세 정보는 작가, 작품, 감성, 색상 정보를 명확히 제공해야 한다.

---

## 10. 포트폴리오 정리 기준

포트폴리오 저장소에서는 실제 운영 정보보다 구조와 구현 의도를 명확히 보여주는 것을 우선한다.

요구사항은 다음과 같다.

- 민감정보는 제거한다.
- 원본 사업계획서는 공개하지 않는다.
- 공개용 Markdown 문서로 재작성한다.
- 실행 가능한 코드와 설명 문서를 함께 제공한다.
- 각 앱별 README를 작성한다.
- 최종 README에서 전체 흐름을 요약한다.

---

## 11. 정리

RestArt의 비기능 요구사항은 보안, 성능, 확장성, 유지보수성, 이미지 처리 품질, Unity/AR 파일 관리 기준을 포함한다.

이 기준은 4단계 이후 실제 백엔드, DB, AI, Unity, 웹, 앱 산출물을 정리할 때 적용한다.
